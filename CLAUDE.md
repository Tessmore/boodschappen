## Architecture notes worth knowing up front

- **Svelte 5 runes mode is forced project-wide.** `svelte.config.js` overrides `compilerOptions.runes` to `true` for every file outside `node_modules`. Use `$props()`, `$state()`, `$derived()`, `$effect()` — do not write Svelte 4-style `export let` props or reactive `$:` statements in this repo's code. The existing `+layout.svelte` already uses `$props()`.
- **Adapter is `@sveltejs/adapter-auto`.** No deployment target has been chosen; `adapter-auto` will pick one at build time based on the host. If a real target is needed, swap to a specific adapter rather than relying on detection.
- **Path aliases.** `$lib` → `src/lib` (SvelteKit default). `tsconfig.json` extends `.svelte-kit/tsconfig.json`, which is regenerated by `svelte-kit sync` (run via `prepare` and `check`) — don't hand-edit aliases there; configure them in `svelte.config.js` under `kit.alias` instead.
- **TypeScript is strict** with `checkJs` and `allowJs` on, so any `.js` files added to the project are also type-checked.

## Code style

These are project preferences. They override generic defaults — apply them when writing or editing code here, and call them out when reviewing.

### Naming — names reveal purpose

Classes/types are nouns; functions are verbs; prefer domain terms over abbreviations.

Avoid container names that describe a bag-of-functions instead of a concept:
- `*Manager`, `*Util(s)`, `*Helper`, generic `Common*`, double-suffix like `OrderServiceService`.
- Replace with the domain concept: `Cart` not `CartManager`, `Invoice` not `InvoiceUtils`.

Avoid generic verbs and primitive-flag method names:
- `updateStatus(1)`, `process(...)`, `handle(...)`, `doIt(...)`, `execute(...)`.
- Use the verb the call site actually wants: `user.suspend()`, `cart.add(item)`, `patient.deregister(reason)`.

Avoid placeholder variable names:
- Single letters outside tight loops (`i` in `for (let i ...)` is fine; `u`, `d`, `x` in business logic is not).
- `tmp`, `data`, `info`, `obj` — pick the domain noun (`pendingInvoice`, not `tmp`).

### Small units / composed method

Size thresholds that should trigger a split:
- Backend service / route / controller files > **250 lines** — name the responsibilities and split.
- Svelte components > **200 lines** — extract sub-components or move logic into `$lib`.
- Functions/methods > **40 lines** — apply Composed Method; extract named helpers.

Keep one altitude per function. If a method orchestrates (`validate(); persist(); notify();`) and also contains low-level details (date formatting, fetch config, header munging), extract the low-level steps so the top-level body reads at a single level of abstraction.

### Value objects over primitives

For domain concepts that carry rules, formats, or invariants, prefer a typed value object over a raw `string` / `number` — both as parameters and as DTO/record fields. Candidates: postal codes, phone numbers, money/amounts, date ranges, email addresses, BSN, AGB-codes. A branded type or small class beats a comment that says "this string is actually a BSN."

### Composition over inheritance

Avoid `class X extends Y` chains beyond framework base classes. Prefer a *has-a* role/strategy shape:

```ts
class User {
  constructor(private role: Role) {}
  can(action: Action) { return this.role.allows(action); }
}
```

### Comments — replace *what* with code, keep *why*

If a comment explains **what** the next line does (`// check if user is adult`, `// loop over items`), extract a named function instead (`isAdult(user)`) and delete the comment. Keep comments that explain a non-obvious **why**: a hidden constraint, a workaround for a specific bug, a surprising invariant a future reader would otherwise trip on.
